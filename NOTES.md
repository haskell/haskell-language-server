# Notes

This branch is for experimenting and improving the result of prepare-rename requests.

Currently, `prepareRenameProvider` returns `PrepareRenameDefaultBehavior True` if renaming is possible, and `PrepareRenameDefaultBehavior False` if renaming is not a valid operation.

There are two problems with this:

1. `{defaultBehavior: false}` is not specified in the LSP specification; invalid requests are supposed to result in `null` instead.

2. Not all clients support `defaultBehavior`. The most high-profile such client is the Zed editor (see [this issue](https://github.com/zed-industries/zed/issues/24184)).

## Possible approach

1. Get the HIE AST and look for the smallest AST node at the cursor position. Get all `Name`s contained in this AST node (except for autogenerated ones); also get the span of this node.

2. If no `Name` is found, deem the request invalid and return `Null` (instead of `PrepareRenameDefaultBehavior False`).

3. If at least one `Name` is found, return the span of the AST node as a `Range`.

    The span of the found `Name` would be the wrong thing to return. A `Name` only remembers the span of its *definition site*, so for an imported name, this span would be outside the current module.

## Testing

New test cases and a new test data file have been added. The new tests, named `prepareRenameTests` in `test/Main.hs`, perform prepare-rename requests and verify the returned responses directly using equality assertions, not through golden files.
